2. RAG — Retrieval-Augmented Generation
Q4. What is RAG and why did you use it instead of just using an LLM directly?
Very Important

RAG (Retrieval-Augmented Generation) is a technique that combines a retrieval system (vector search) with a generation system (LLM) to produce factually grounded answers.

If I send a question directly to an LLM like Llama without any context, it will answer from its training data — which may be outdated, wrong, or hallucinated. With RAG:

  • I first retrieve the most relevant pieces from the user's own document
  • Then pass that exact text as context to the LLM
  • The LLM generates answers based on that specific content

This eliminates hallucinations and makes answers traceable to real sources.

Q5. Explain the full RAG pipeline in your project step by step.
Must Know
  1. Upload: User uploads a PDF/TXT file
  2. Chunking: File is split into ~500-character overlapping chunks
  3. Embedding: Each chunk is converted to a 384-dimension vector using MiniLM
  4. Storage: Vectors + metadata are upserted into Endee vector DB
  5. Query: User types a question → it is also embedded into a vector
  6. Retrieval: Endee performs cosine similarity search → returns Top-K relevant chunks
  7. Generation: Chunks + question are sent to Groq LLM as a prompt
  8. Response: LLM generates a formatted, sourced answer displayed in the UI
Q6. What is the difference between RAG and fine-tuning an LLM?
  • Fine-tuning: Retraining the LLM on custom data. Expensive, slow, requires GPUs, and the knowledge is "baked in" permanently.
  • RAG: No model training needed. You just retrieve relevant text at query time and inject it into the prompt. Easy to update — just add/remove documents from the vector DB.

RAG is perfect when your data changes frequently (like company documents, reports, etc.). Fine-tuning is better for teaching the model a new style or behavior.

3. Endee Vector Database
Q7. What is the Endee vector database and how did you use it?
Must Know

Endee is a lightweight, open-source vector database designed specifically for AI applications. It stores high-dimensional number arrays (vectors) and allows lightning-fast similarity searches using HNSW (Hierarchical Navigable Small World) graph indexing.

In my project, I used Endee to:

  • Create an index with dimension=384 and space_type="cosine"
  • Store document chunk vectors along with metadata (source file, chunk index, doc_id, text)
  • Query the index with a question vector to retrieve the top-K most similar chunks
Q8. Why Endee over other vector databases like Pinecone, Weaviate, or ChromaDB?
  • It was the assignment requirement to use Endee specifically
  • Endee is self-hostable via Docker — no cloud account needed
  • It has a clean Python SDK that is easy to integrate
  • It's lightweight and fast for local development and demos
  • Being open-source, it gives full control — no vendor lock-in
Q9. How did you run and configure Endee?

Endee is run as a Docker container. I built the Docker image locally from the Endee GitHub repository's Dockerfile and then used docker-compose.yml to start it:

docker compose up -d   # Starts Endee on http://localhost:8080

The Python backend then connects to it at http://localhost:8080/api/v1 using the Endee Python SDK.

4. Vector Embeddings & Semantic Search
Q10. What is a vector embedding?
Important

A vector embedding is a list of numbers (a vector) that represents the semantic meaning of a piece of text. Text with similar meaning will have vectors that are mathematically close to each other in high-dimensional space.

For example: "What is AI?" and "Define Artificial Intelligence" would produce very similar vectors, even though the words are different.

In my project, I use the all-MiniLM-L6-v2 model to convert any text into a 384-dimensional vector.

Q11. What is cosine similarity and why is it used for vector search?

Cosine similarity measures the angle between two vectors, not their magnitude. It gives a score between -1 and 1 where:

  • 1.0 = identical meaning
  • 0.0 = unrelated
  • -1.0 = opposite meaning

It's preferred for NLP because the direction of the vector matters more than its size. Two documents about the same topic will point in the same direction even if one is shorter.

Q12. Why did you choose the MiniLM-L6-v2 embedding model?
  • It is free and open-source — runs entirely locally, no API cost
  • It's fast and lightweight — 22M parameters, perfect for a demo
  • It produces high-quality 384-dim embeddings — good accuracy for semantic similarity
  • It works well on short to medium length text chunks
Q13. What chunking strategy did you use and why?

I split documents into ~500 character chunks with overlap. Chunking is necessary because:

  • LLMs have a limited context window — you can't pass entire large documents
  • Smaller chunks produce more precise embeddings
  • Overlap ensures that important information at chunk boundaries is not lost
5. Backend — FastAPI Architecture
Q14. Why FastAPI and not Flask or Django?
  • Async support — FastAPI is built on ASGI and handles concurrent requests efficiently
  • Automatic API docs — Swagger UI is auto-generated at /docs
  • Pydantic validation — Request/response models are automatically validated
  • Performance — FastAPI is one of the fastest Python web frameworks
Q15. Explain the key API endpoints in your backend.
  • POST /upload — Accepts a file, chunks it, embeds it, stores in Endee
  • POST /ask — Full RAG pipeline: embed → retrieve → LLM → answer
  • POST /search — Pure vector search, returns raw chunks with scores
  • GET /health — Verifies both the API server and Endee are reachable
  • GET /documents — Lists all indexed documents from Endee
  • DELETE /documents/{id} — Removes all chunks for a document
Q16. How does your backend serve the frontend?

The FastAPI backend serves the HTML/CSS/JS frontend as static files using StaticFiles mounted at the root path /. This means the entire application — both API and UI — is accessible from a single server on port 8000, without needing a separate web server like Nginx.

6. LLM Integration — Groq & Llama
Q17. What is Groq and why did you use it?

Groq is an AI inference platform that runs open-source LLMs at extremely high speeds using their custom LPU (Language Processing Unit) hardware. I used Groq because:

  • It provides free API access to Llama 3, Mixtral, and other models
  • Response speeds are 5-10x faster than standard GPU inference
  • Simple, clean API — compatible with OpenAI's API format
Q18. What is a system prompt and how did you use it?

A system prompt is an instruction sent to the LLM before the user's question. It sets the behavior, tone, and constraints for the model.

In my project, the system prompt tells the LLM to:

  • Answer only using the provided document context (no hallucinations)
  • Format the response in Markdown (headers, bullet points, code blocks)
  • Cite sources and be concise
Q19. What happens if the LLM doesn't have a Groq API key?

I built a graceful fallback mechanism. If no API key is configured, instead of crashing, the system returns the raw retrieved document chunks as the "answer" — still providing useful information to the user, just without LLM summarization.

7. Frontend — UI & Interaction
Q20. How does your frontend render the AI responses so nicely?

The AI responses are returned as Markdown-formatted text. The frontend uses:

  • marked.js — Parses Markdown into HTML (bold, headers, lists, code blocks)
  • highlight.js — Applies syntax highlighting to code blocks (dark theme, like VSCode)

The frontend supports both click-to-browse and drag-and-drop file uploads. When a file is selected, it is sent to the POST /upload endpoint as a multipart/form-data request. An animated progress bar provides visual feedback, and a toast notification confirms success or shows an error.

8. Challenges & Problem Solving
Q22. What was the biggest technical challenge you faced?
Great for leaving an impression

The biggest challenge was a bug in the Endee Python SDK. The SDK internally called .get() on a Pydantic model object (which doesn't support that method), causing a 'VectorItem' object has no attribute 'get' error during document upsert.

To fix this without modifying the SDK's source, I applied a monkey-patch — dynamically adding a .get() method to the VectorItem class at runtime:

VectorItem.get = lambda self, key, default=None: getattr(self, key, default)

This demonstrates understanding of Python's dynamic nature and SDK internals.

Q23. How did you handle the decommissioned Groq model?

When I first tested the project, Groq returned an error that the llama3-8b-8192 model had been decommissioned. I investigated the error log, checked the Groq deprecation docs, and updated both the .env configuration file and the llm_client.py default to use llama-3.1-8b-instant, which is the current supported replacement.

Q24. What happens when someone asks about something not in the documents?

The system is designed to be honest. The LLM receives a strict system prompt instructing it to say "The provided context does not contain information about this topic" rather than making up an answer. The retrieved chunks with low relevance scores also signal to the model that the matched content is not highly related.

9. General AI/ML Concepts
Q25. What are LLM hallucinations and how does RAG help?

Hallucinations are when an LLM confidently generates false or made-up information. This happens because LLMs are trained to produce statistically likely text — not necessarily factual text.

RAG helps because the LLM's context is constrained to retrieved real document chunks. The system prompt reinforces that it should only use that context. This grounds the answer in verified source material.

Q26. What is HNSW and how does it make vector search fast?

HNSW (Hierarchical Navigable Small World) is a graph-based approximate nearest neighbor search algorithm. Instead of comparing a query vector against every stored vector (O(n) brute force), HNSW builds a hierarchical graph structure that allows search in O(log n) time — extremely fast even with millions of vectors.

Endee uses HNSW internally for its vector indexing.

Q27. What is a context window in an LLM?

A context window is the maximum amount of text an LLM can process in a single request (measured in tokens, roughly ~4 characters per token). For example, Llama 3.1-8b has an 8,192 token context window.

This is why chunking is important — you can't send an entire 100-page document to an LLM. Instead, you retrieve only the relevant chunks (typically 3-10) to fit within this limit.
